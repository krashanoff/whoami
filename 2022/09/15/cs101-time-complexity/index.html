Time complexity is the fundamental metric in computer science used to describe the runtime of an algorithm. If an algorithm for a given task has a lower time complexity than another for the same task, then it is provably faster for all possible inputs. The basis for time complexity lay in the growth rate of the function describing the exact runtime of the algorithm in question.

Given:

  • An array of numbers, AAA. For example, A=[1,3,2]A = [ 1, 3, 2 ]A=[1,3,2].
  • An empty array, RRR.

We could write an algorithm to sort the array AAA in order of least to greatest.

R = an empty array

while A is not empty:
	MINIMUM = first element of A
	for each number, N, in A:
		if N < MINIMUM then let MINIMUM = N
		otherwise, do nothing

	insert MINIMUM at the start of R
	remove MINIMUM from A

How much time does it take for this algorithm to execute? Let's say it takes one second (111) to check an element, and ∣A∣|A|∣A∣, the length of our input array, is nnn.

To fill up RRR, we have to find every minimum element of AAA. We have to check all nnn items of AAA to find the first minimum (all items), then n−1n-1n−1 elements (we already found a minimum), then n−2n-2n−2 (we found another), ... and so on. So it takes n∗1=nn * 1 = nn∗1=n time to find a minimum in the worst case scenario.

In the absolute worst case scenario, the array AAA is in reverse order. In this situation, we have to check the entire array to find every "next minimum" element to insert into RRR. Since we need to locate nnn elements to fill RRR, and we have to check at most nnn items to get each of those, we say this takes O(n2)O(n^2)O(n​2​​) operations, which means the algorithm's runtime is always less than some constant multiple of n2n^2n​2​​. For example, 2n22n^22n​2​​ or 10n210n^210n​2​​ are all multiples of n2n^2n​2​​, but since there exists some integer greater than their current multiple, they are all O(n2)O(n^2)O(n​2​​).

Intuitively, this might not make sense. "Well couldn't you just say O(n)O(n)O(n) is nnn times a billion?" Yes, you could, but if your algorithm takes n2n^2n​2​​ steps in the worst case scenario, then for some value of nnn, it will pass "nnn times a billion":

"Aha! I got you. See, nnn times a billion is higher than your n2n^2n​2​​!" Not so fast. What if we scale to, say, 101810^{18}10​18​​?"

Since the function n2n^2n​2​​ grows faster than nnn, there is a point where the algorithm could hypothetically take longer than "a billion times nnn steps". Since the upper bound is all that really matters here, programmers use this to denote the upper bound of an algorithm's runtime. We call this "time complexity".

Appendix

Unverified Python

Here's the algorithm mentioned above in Python. No guarantees of correctness, lol.

A = [1, 3, 2]
R = []

while A:
	minimum = A[0]
	for N in A:
		minimum = min(minimum, N)
	R.append(minimum)
	A.remove(minimum)

Some considerations:

  • There is already a more efficient sorting algorithm implemented in Python, callable with sort().
  • This removes elements from the list A. If you wish to retain elements, a copy would need to be made.
