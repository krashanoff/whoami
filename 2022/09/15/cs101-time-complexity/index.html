CS101: Time Complexity

This post was originally written and posted to a Discord server shared by my friends and I on September 1st, 2022. I made minor edits between then and the current post.

Time complexity is the fundamental metric in computer science used to describe the runtime of an algorithm. If an algorithm for a given task has a lower time complexity than another for the same task, then it is provably faster for all possible inputs. The basis for time complexity lay in the growth rate of the function describing the exact runtime of the algorithm in question.

Given:

  • An array of numbers, AA. For example, A=[1,3,2]A = [ 1, 3, 2 ].
  • An empty array, RR.

We could write an algorithm to sort the array AA in order of least to greatest.

R = an empty array

while A is not empty:
  MINIMUM = first element of A
  for each number, N, in A:
    if N < MINIMUM then let MINIMUM = N
    otherwise, do nothing

  insert MINIMUM at the start of R
  remove MINIMUM from A

How much time does it take for this algorithm to execute? Let’s say it takes one second (11) to check an element, and |A||A|, the length of our input array, is nn.

To fill up RR, we have to find every minimum element of AA. We have to check all nn items of AA to find the first minimum (all items), then n−1n-1 elements (we already found a minimum), then n−2n-2 (we found another), … and so on. So it takes n*1=nn * 1 = n time to find a minimum in the worst case scenario.

In the absolute worst case scenario, the array AA is in reverse order. In this situation, we have to check the entire array to find every “next minimum” element to insert into RR. Since we need to locate nn elements to fill RR, and we have to check at most nn items to get each of those, we say this takes O(n2)O(n^2) operations, which means the algorithm’s runtime is always less than some constant multiple of n2n^2. For example, 2n22n^2 or 10n210n^2 are all multiples of n2n^2, but since there exists some integer greater than their current multiple, they are all O(n2)O(n^2).

Intuitively, this might not make sense. “Well couldn’t you just say O(n)O(n) is nn times a billion?” Yes, you could, but if your algorithm takes n2n^2 steps in the worst case scenario, then for some value of nn, it will pass “nn times a billion”:

“Aha! I got you. See, nn times a billion is higher than your n2n^2!” Not so fast. What if we scale to, say, 101810^{18}?”

Since the function n2n^2 grows faster than nn, there is a point where the algorithm could hypothetically take longer than “a billion times nn steps”. Since the upper bound is all that really matters here, programmers use this to denote the upper bound of an algorithm’s runtime. We call this “time complexity”.

1 Appendix

1.1 Unverified Python

Here’s the algorithm mentioned above in Python. No guarantees of correctness, lol.

A = [1, 3, 2]
R = []

while A:
  minimum = A[0]
  for N in A:
    minimum = min(minimum, N)
  R.append(minimum)
  A.remove(minimum)

Some considerations:

  • There is already a more efficient sorting algorithm implemented in Python, callable with sort().
  • This removes elements from the list A. If you wish to retain elements, a copy would need to be made.
